01 Project Overview

Vision & Objectives

IntellMeet is a full-stack, AI-powered video conferencing and meeting intelligence platform built for enterprise teams, startups, and distributed organisations. The central problem it solves is straightforward: the knowledge from most meetings evaporates the moment the call ends.

IntellMeet addresses this by combining live video infrastructure (Stream SDK), a large-language-model AI pipeline (Google Gemini 2.5 Flash), and a modern team collaboration layer — so every meeting automatically produces a structured summary, extracted action items, and a searchable transcript, with zero effort from participants.

The project was developed as part of Zidio Development and is designed to be benchmarked against commercial products such as Zoom, Google Meet, Linear, Loom, and Notion.

Target Users & Use Cases

  • Enterprise teams that need meeting accountability and documentation at scale.
  • Startups running daily standups, sprint reviews, and asynchronous follow-ups.
  • Remote-first organisations spread across time zones who need reliable written records.
  • Managers and team leads tracking task completion derived from meetings.
  • Teams already using Slack or Notion who want meeting intelligence pushed into existing workflows.

Business Value Delivered

  • Zero-effort documentation. Meeting notes, summaries, and action items are generated automatically — no manual note-taking required.
  • Accountability at scale. Action items carry assignee and due-date metadata and are promoted directly into a managed Kanban task board.
  • Knowledge retention. Full transcripts, AI summaries, and recordings remain searchable long after a meeting ends.
  • Ecosystem integration. Slack and Notion sync pushes meeting intelligence into the tools teams already rely on.
  • Executive visibility. An analytics dashboard surfaces meeting cadence, engagement, and productivity trends.

Architecture Overview

IntellMeet is structured as a two-tier web application: a React single-page application hosted on Vercel's CDN communicates with a Node.js/Express REST API hosted on Render. Both layers are protected by Clerk for authentication and authorisation.

Data Flow

  1. A user authenticates via Clerk and starts or joins a meeting through the React frontend.
  2. The frontend calls the Express API, which creates a Stream call and returns a scoped user token.
  3. During the call, Stream's media servers handle WebRTC video, in-meeting chat, live closed captions, and continuous transcription.
  4. When the call ends, Stream fires a call.transcription_ready webhook to the Express backend.
  5. The webhook controller retrieves the transcript from Stream's storage, persists it to MongoDB, and triggers the Gemini AI summarisation pipeline.
  6. If the webhook is delayed or missed, an Inngest background job polls for pending transcripts and re-triggers processing after a 90-second wait.
  7. The Gemini-generated summary and action items are persisted to MongoDB and surfaced on the team dashboard.
  8. From the dashboard, action items can be promoted to the Kanban board, exported as a PDF, or pushed to Slack and Notion.

06 Technical Highlights

Security Measures

MeasureImplementation DetailOWASP Category
JWT Authentication Every API route protected by clerkMiddleware(). Tokens verified server-side; no client-side trust is assumed. A01 — Broken Access Control
Environment Secrets All API keys stored in .env files, excluded from version control via .gitignore. No secrets are committed to the repository. A02 — Cryptographic Failures
Mongoose Schema Validation All data models enforce type constraints, required fields, and enum values — preventing malformed or injected documents from reaching the database. A03 — Injection
CORS Whitelist Backend restricts cross-origin requests to the known frontend URL defined in the CLIENT_URL environment variable. No wildcard origins are permitted. A05 — Security Misconfiguration
Stream Token Scoping User tokens for Stream are generated server-side with limited call scopes. Clients never hold Stream API secrets directly. A07 — Identification & Auth Failures
Webhook Signature Verification Incoming Stream webhooks are verified using the Stream signing secret before any transcript or recording data is processed. A08 — Software & Data Integrity Failures

Performance & Scalability

  • TanStack Query caching. API responses are cached client-side with a stale-while-revalidate strategy, eliminating redundant network calls on navigation.
  • Granular Zustand stores. Lightweight client-state slices for meetings, teams, and tasks avoid unnecessary re-renders by keeping state updates targeted.
  • Route-level code splitting. React Router's dynamic import() keeps the initial bundle lean, improving time-to-interactive for first-time visitors.
  • Async AI processing via Inngest. Summarisation runs in the background, allowing the API to respond to Stream webhooks immediately and avoid timeout errors.
  • MongoDB indexing. Collections are indexed on sessionId, teamId, and clerkId fields for fast lookups on the most common query patterns.
  • pdfkit streaming. PDFs are streamed directly to the HTTP response pipe rather than buffered in memory, allowing the server to handle large documents without memory pressure.

Challenges & Solutions

#ChallengeSolution Applied
1 Webhook timing. Stream's transcription_ready webhook sometimes arrives 30–120 s after call end, causing the AI pipeline to run on missing data. Inngest fallback job: a call.ended event schedules a background function that waits 90 s, checks whether the transcript has arrived, and re-triggers summarisation if not.
2 Gemini JSON instability. Despite setting responseMimeType: "application/json", Gemini occasionally wraps output in markdown code fences. Double-layer parsing: first attempt JSON.parse() directly on the response; fall back to regex extraction /\{[\s\S]*\}/ if that fails. An error is thrown only if neither approach succeeds.
3 Adaptive video layout. Replicating Google Meet's dynamic grid for varying participant counts and screen-share states required complex layout logic. Custom React component using CSS Grid with a participant-count-based column formula and a dedicated screen-share lane that activates automatically when sharing begins.
4 Render cold starts. Free-tier Render instances spin down after 15 min of inactivity, causing 30+ second delays on the first request. Added a public GET /health endpoint for monitoring. Cold-start limitation is documented; migration to Railway or Fly.io is listed on the roadmap.
5 Clerk–Stream identity mapping. Clerk and Stream use separate user ID systems; keeping them in sync across meetings, recordings, and transcripts required a custom mapping layer. A resolveParticipants utility on the backend cross-references Clerk user records against Stream call participant data to produce a unified participant list for each session.

09 Personal Reflection & Roadmap

Key Learnings

Engineering Insights

  • Event-driven design matters. Building around webhooks (Stream → backend → Inngest) taught resilient async system design that never blocks the happy path.
  • AI output is non-deterministic. Working with Gemini's JSON mode revealed that LLMs require defensive parsing — a fallback regex layer is essential.
  • WebRTC surface area is large. The Stream SDK abstracts media negotiation, but debugging layout reflow on participant changes required deep understanding of React rendering lifecycles.

Best Practices Applied

  • Separation of concerns. A dedicated services layer (ai.service.js, integrations.service.js) cleanly separates business logic from HTTP controllers.
  • 12-Factor App principles. Config via environment variables; stateless processes; backing services as attached resources.
  • Idempotent operations. summarizeAndPersist wipes previous action items before reinserting — safe to re-run without producing duplicate data.
  • Graceful degradation. AI pipeline failures set aiProcessingStatus: "failed" without affecting the call experience.
